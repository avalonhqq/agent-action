# 设计决策记录

## ADR-001：采用一个主项目持续迭代

- 日期：2026-07-14
- 状态：已确认
- 决策：12 周围绕 BiliSupport AI 持续演进，不创建互不相关的周项目。
- 原因：可以形成完整系统认知、可演示成果和真实重构经验。

## ADR-002：知识库是核心，但最终交付是完整客服网站

- 状态：已确认
- 决策：保留知识 ingestion/RAG 的深度，同时补充 LLM/SSE、会话、Mock 业务工具、网站、安全、评估和部署。
- 原因：仅有知识库内核不足以体现企业客服端到端能力。

## ADR-003：使用 LangGraph 确定性工作流

- 状态：已确认
- 决策：Supervisor 和专业 Agent 在显式状态、节点和条件边中运行，不使用完全自由的 ReAct 主流程。
- 原因：客服要求可回放、可测试、有限步、可降级和权限可控。

## ADR-004：业务系统使用仿真数据和受控工具

- 状态：已确认
- 决策：会员、订单、稿件、处罚、工单均为 Mock/仿真服务；Agent 不执行任意 SQL。
- 原因：保证安全、可复现，并保持未来真实 API 替换边界。

## ADR-005：人工客服仅实现技能组接口

- 状态：已确认
- 决策：实现转接判断、技能组、交接包和工单编号，不实现坐席调度与 Redis 排队。
- 原因：将学习时间集中在 AI 应用工程和可靠性治理。

## ADR-006：改用标准 src 布局和 pyproject

- 状态：已执行
- 决策：代码位于 `src/bili_support`，依赖和质量工具集中在 `pyproject.toml`。
- 原因：避免导入歧义，贴近可发布 Python 项目的工程实践。
