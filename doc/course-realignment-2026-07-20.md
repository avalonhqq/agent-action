# 课程重排记录：大模型专项模式

> 日期：2026-07-20

## 1. 调整原因

学习者明确只关注大模型相关能力。原计划中 FastAPI、数据库、迁移、CRUD、鉴权、页面和部署占用较多学习时间，但这些内容不是当前学习目标。因此保留完整商业项目要求，同时改变学习分工，而不是删除工程能力。

## 2. 已自动完成并通过校验的底座

第 1～3 周已经完成：

- 工程、配置、错误边界、日志、探针和质量门禁。
- LLM Provider、Prompt、结构化输出、SSE、上下文和 Usage。
- SQLAlchemy、Alembic、用户/会话/消息/模型调用。
- Repository/Service、简单鉴权、持久化 SSE、NiceGUI 和 Compose。

MySQL/Redis 接入后的验证结果：Ruff、strict mypy、pre-commit 通过，pytest 106 项通过，Alembic 重复升级与真实 MySQL/Redis 应用链路通过。

## 3. 旧计划到新计划映射

| 旧安排                | 新安排                     | 调整理由                            |

## 4. 后续自动化承诺

每个 AI 模块开始前，Codex 自动完成必要的模型表、迁移、CRUD、管理 API、页面、Fixture 和部署接线，并提供：

1. 改动文件清单。
2. 接口与数据契约。
3. 单元和集成测试。
4. Ruff、strict mypy、pytest、迁移及专项验证结果。
5. 已知边界和未使用真实外部服务的说明。

工程底座通过后，学习讨论才进入 Prompt、Schema、Chunk、RetrievalPolicy、Graph 或 Agent 实验。

## 5. 学习者不再承担

- 手写重复 CRUD 和分页。
- 编写普通数据库迁移和 Repository。
- 接线通用鉴权、中间件、日志和 Docker。
- 为了课程进度重复实现管理页面。

学习者仍需理解这些边界如何影响模型安全、上下文、权限、延迟和可观测性，但无需以它们作为主要编码作业。

## 6. 下一课

第 4 周“意图识别与结构化决策”：先建立 Intent Schema、分类 Prompt、结构化输出和评估基线，再让该决策驱动后续检索与 Agent 路由。
