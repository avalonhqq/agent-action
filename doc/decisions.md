# 设计决策记录

## ADR-001：采用一个主项目持续迭代

- 日期：2026-07-14
- 状态：已确认
- 决策：12 周围绕 BiliSupport AI 持续演进，不创建互不相关的周项目。
- 原因：可以形成完整系统认知、可演示成果和真实重构经验。

## ADR-002：知识库是核心，但最终交付是完整客服网站

- 状态：已确认
- 决策：保留知识 ingestion/RAG 的深度，同时补充 LLM/SSE、会话、Mock 业务工具、网站、安全、评估和部署。
- 原因：仅有知识库内核不足以体现企业客服端到端能力。

## ADR-003：使用 LangGraph 确定性工作流

- 状态：已确认
- 决策：Supervisor 和专业 Agent 在显式状态、节点和条件边中运行，不使用完全自由的 ReAct 主流程。
- 原因：客服要求可回放、可测试、有限步、可降级和权限可控。

## ADR-004：业务系统使用仿真数据和受控工具

- 状态：已确认
- 决策：会员、订单、稿件、处罚、工单均为 Mock/仿真服务；Agent 不执行任意 SQL。
- 原因：保证安全、可复现，并保持未来真实 API 替换边界。

## ADR-005：人工客服仅实现技能组接口

- 状态：已确认
- 决策：实现转接判断、技能组、交接包和工单编号，不实现坐席调度与 Redis 排队。
- 原因：将学习时间集中在 AI 应用工程和可靠性治理。

## ADR-006：改用标准 src 布局和 pyproject

- 状态：已执行
- 决策：代码位于 `src/bili_support`，依赖和质量工具集中在 `pyproject.toml`。
- 原因：避免导入歧义，贴近可发布 Python 项目的工程实践。

## ADR-007：吸收 Small-to-Big 与策略化混合检索

- 日期：2026-07-15
- 状态：已确认
- 参考：[enterprise-rag-financial-reports](https://github.com/stevenfight/enterprise-rag-financial-reports)
- 决策：引入 Child 召回/Parent 上下文、中文领域词典、批量 Rerank、意图驱动 RetrievalPolicy、多实体 Evidence Coverage 和检索调试链路。
- 原因：这些能力能改善客服短问法召回、完整条款上下文、比较问题覆盖和检索可解释性。
- 约束：不复制金融硬编码、内存会话、公开思维链、同步阻塞调用、单体检索文件和未经评估的固定阈值。

## ADR-008：核心异常与 HTTP 框架解耦

- 日期：2026-07-17
- 状态：已执行
- 决策：业务层抛出带稳定错误码的 `AppError`，FastAPI 边界统一映射 HTTP 响应；未知异常只返回通用错误。
- 原因：避免业务代码依赖 Web 框架，并防止内部异常、输入值和堆栈泄露到客户端。

## ADR-009：Request ID 贯穿响应与结构化日志

- 日期：2026-07-17
- 状态：已执行
- 决策：接受格式安全的调用方 `X-Request-ID`，否则生成新 ID；响应头、错误体和访问日志使用同一 ID。
- 原因：支持跨网关和后续 Agent/Tool 链路排障，同时避免记录查询参数和请求体中的敏感值。

## ADR-010：健康探针保持轻量，就绪探针表达依赖状态

- 日期：2026-07-17
- 状态：已执行
- 决策：`/health` 只证明进程存活；`/ready` 检查当前强制依赖。第一周只有已在启动时校验的配置。
- 原因：依赖故障不应触发错误的进程重启；后续数据库和模型成为强制依赖时扩展 readiness checks。

## ADR-011：模型供应商位于内部 Provider 边界之后

- 日期：2026-07-19
- 状态：已执行
- 决策：业务只依赖 `LLMProvider` 和项目内部 Pydantic 类型；Mock 与 OpenAI-compatible 实现由配置选择。
- 原因：隔离第三方协议变化，支持无密钥开发、确定性测试和后续供应商替换。

## ADR-012：Prompt 版本和结构化原因码进入观测数据

- 日期：2026-07-19
- 状态：已执行
- 决策：模型调用记录 Prompt 版本、Token、耗时、状态和稳定错误码；不记录用户全文、模型全文或私有思维链。
- 原因：满足回归、排障和成本分析，同时坚持数据最小化并减少敏感内容泄露。

## ADR-013：重试只覆盖暂时性失败

- 日期：2026-07-19
- 状态：已执行
- 决策：仅对网络传输错误、429 和指定 5xx 做有限指数退避；普通 4xx 和非法响应不重试；流已输出后不从头重放。
- 原因：避免无效费用、延迟放大、服务雪崩与流式文本重复。

## ADR-014：Query Rewrite 采用可解释的保守基线

- 日期：2026-07-19
- 状态：已执行
- 决策：只对高置信度实体指代执行确定性替换，否则保留原查询并返回结构化原因码。
- 原因：错误改写会改变检索意图；先建立可测试基线，再由后续 Golden Dataset 证明模型改写是否值得引入。

## ADR-015：本地 SQLite 与部署 PostgreSQL 共用异步数据边界（已被 ADR-020 取代）

- 日期：2026-07-19
- 状态：已取代
- 决策：本地零依赖开发默认 SQLite + aiosqlite，Compose 使用 PostgreSQL + asyncpg；统一通过 SQLAlchemy 2 与 Alembic 管理。
- 原因：降低学习门槛，同时保留真实部署数据库、事务和迁移经验。

## ADR-016：模型网络调用之外使用短数据库事务

- 日期：2026-07-19
- 状态：已执行
- 决策：先用短事务保存用户消息，模型完成或流结束后再用短事务保存助手消息和调用审计，不让事务跨越 LLM 网络等待。
- 原因：模型失败时不丢问题，并避免长事务占用连接和锁。

## ADR-017：模型调用使用消息外键形成审计链

- 日期：2026-07-19
- 状态：已执行
- 决策：ModelCall 同时关联 conversation、user_message 和可选 assistant_message；Request ID 作为追踪字段但不假设全局唯一。
- 原因：调用方可能重复 Request ID，显式外键才能稳定回答一次调用由哪条消息触发并生成哪条结果。

## ADR-018：第三周认证是可替换演示边界

- 日期：2026-07-19
- 状态：已执行
- 决策：本周使用共享 Bearer Token 加受限用户标识；生产化必须替换为 OIDC/OAuth2/企业 SSO，并从已验证 claims 获取身份。
- 原因：满足学习阶段鉴权和用户隔离目标，但不夸大共享秘密的身份保证能力。

## ADR-019：课程调整为大模型专项模式

- 日期：2026-07-20
- 状态：已确认
- 决策：学习者只重点学习 Prompt、结构化决策、RAG、意图、Agent、安全与评估；CRUD、迁移、鉴权、页面、部署等由 Codex 自动实现、测试和留档。
- 原因：减少通用 Web 工程对学习时间的占用，把实验和讨论集中在影响模型效果、可靠性与可解释性的决策上。
- 约束：自动完成的代码仍必须通过 Ruff、strict mypy、pytest、迁移和相关专项验收；不得用 Mock 指标冒充真实模型效果。

## ADR-020：MySQL 作为事实存储，Redis 只缓存模型历史

- 日期：2026-07-20
- 状态：已执行
- 决策：本地和 Compose 使用 MySQL 8 + aiomysql 保存业务事实，Redis asyncio 缓存有 TTL 的模型可见会话历史；测试继续使用 SQLite。
- 原因：匹配当前可用本地基础设施，同时保持异步 SQLAlchemy 边界和快速测试能力。
- 约束：Redis 不是消息事实来源；缓存异常回退 MySQL；生产必须使用最小权限 MySQL 用户和受保护的 Redis。
