# 每周质量门禁

学习进度主要由核心能力是否实现、能否运行、能否解释和是否留档决定。测试用例不再作为进入下一
步骤或下一周的阻塞门禁；必要测试由 Codex 作为后台工程质量保障自动补充或运行，并记录结果。

## 1. 可运行

- 本周主流程能够实际演示。
- 无外部密钥时有明确 Mock 或降级方式。
- README 包含启动和验证命令。

## 2. 可验证（非阻塞）

- 新增核心逻辑优先使用单元测试或可重复检查验证，但缺少测试用例不阻塞学习进度。
- 跨层链路由 Codex 按风险补充集成验证。
- AI 效果修改有固定样本或评估数据。

## 3. 可解释

- 能解释本周技术选型和备选方案。
- 能说明失败场景与降级路径。
- 能指出当前实现的商业化边界。

## 4. 可维护

- `ruff check .` 和核心模块 mypy 用于及时发现明显工程错误。
- 模块职责清晰，不把 SQL、Prompt、HTTP 混在一个函数。
- 配置和密钥不写死。

## 5. 有留档

- 更新 `progress.md`。
- 记录遇到的问题、根因和修复。
- 更新架构/API/安全文档中受影响部分。
- 保存本周演示用例和测试结果。

## 自动底座门禁

标记为“自动底座”的任务由 Codex 负责以下后台质量保障，不作为学习者的推进门禁：

- Codex 给出实现清单和影响范围。
- 按风险补充单元/集成验证，迁移保持可重复执行。
- 运行 Ruff、strict mypy、pytest 和 pre-commit，并如实记录未通过项。
- 不把基础设施 Mock 的通过率当作大模型效果指标。
- AI 实验开始前记录底座版本，实验期间原则上保持底座不变。
