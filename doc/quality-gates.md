# 每周质量门禁

每周结束必须同时满足以下五类条件，缺一项则不进入下一周。

## 1. 可运行

- 本周主流程能够实际演示。
- 无外部密钥时有明确 Mock 或降级方式。
- README 包含启动和验证命令。

## 2. 可测试

- 新增核心逻辑有单元测试。
- 跨层链路至少有一个集成测试。
- AI 效果修改有固定样本或评估数据。

## 3. 可解释

- 能解释本周技术选型和备选方案。
- 能说明失败场景与降级路径。
- 能指出当前实现的商业化边界。

## 4. 可维护

- `ruff check .` 通过。
- `mypy` 核心模块通过。
- 模块职责清晰，不把 SQL、Prompt、HTTP 混在一个函数。
- 配置和密钥不写死。

## 5. 有留档

- 更新 `progress.md`。
- 记录遇到的问题、根因和修复。
- 更新架构/API/安全文档中受影响部分。
- 保存本周演示用例和测试结果。

## 自动底座门禁

标记为“自动底座”的任务也必须满足：

- Codex 给出实现清单和影响范围。
- 新增基础代码有单元/集成测试，迁移可重复执行。
- Ruff、strict mypy、pytest 和 pre-commit 全部通过。
- 不把基础设施 Mock 的通过率当作大模型效果指标。
- AI 实验开始前记录底座版本，实验期间原则上保持底座不变。
